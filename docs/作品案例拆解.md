# 视觉营销工作流发现 Agent：作品案例拆解

## 产品定位

目标用户是正在寻找 AI 落地机会的产品经理和业务负责人。真正困难的不是生成内容，而是识别哪一段人工流程值得 Agent 介入，并解释为什么。

## 用户问题与产品设计

- 用户缺少可信输入：系统先规划搜索问题，再从公开证据或内置证据池抽取事实。
- 候选太多无法比较：用统一维度为工作流评分，并保留证据链接。
- AI 容易越过业务判断：候选工作流必须经过 human lock 才能进入深度设计。
- 方案难以复盘：每个节点写入 trace、metrics、health 和模型用量。
- 现场网络不稳定：实时搜索失败时切换证据池，并在结果中暴露 source_mode。

## 核心流程

搜索规划 -> 证据抽取 -> 候选工作流 -> 多维评分 -> 人工锁定 -> 痛点拆解 -> Agent 介入设计 -> 风险审查 -> 报告导出。

## AI 机制

AI 不直接替用户决定做哪个 Agent。它负责归纳证据、形成候选、解释评分和生成结构化方案。原始证据、选择结果和节点输出都可追溯。模型失败时，系统保留规则与证据驱动的可运行路径。

## 技术架构

- 表现层：Python ThreadingHTTPServer 与静态 Web 工作台。
- 编排层：默认状态机，可切换 LangGraph。
- 业务节点：query planner、search executor、evidence builder、candidate scorer、human lock、deep dive、risk reviewer、reporter。
- 可观测性：run_id、request_id、trace.json、metrics.json、health.json、llm_usage.json。
- 模型层：StepFun OpenAI-compatible API，记录 provider usage 与错误。

## 关键取舍

- 第一版不依赖 PostgreSQL 或外部可观测平台，降低现场演示风险。
- 搜索接口与下游评分解耦，替换搜索供应商不影响业务编排。
- 保留状态机 fallback，LangGraph 是编排增强而不是唯一运行条件。

## 验收证据

- 健康检查可访问。
- discovery 返回候选和证据。
- deep-dive 输出流程图、风险审查和最终报告。
- 真实模型运行必须出现 stepfun_real_api 和 provider usage。

## 两分钟讲法

这个项目解决的是 AI Agent 最前面的机会发现问题。我先把工作流发现拆成搜索、证据、候选、评分和人工锁定，再在锁定后生成 Agent 化方案。技术上我用状态机和 LangGraph 两种编排，所有节点都带 trace、metrics 和模型用量。它的重点不是生成得快，而是每个结论都能回到证据和决策过程。

## 五分钟展开

先讲为什么需要工作流发现，再展示候选比较和 human lock。随后打开 trace 与报告，解释搜索失败回退、模型结构化输出、节点级可观测性和 LangGraph 替换编排层的设计。

## 高频追问

- 为什么需要 human lock：候选评分是辅助判断，业务选择不能交给模型自动完成。
- 为什么保留两种 runner：现场稳定性与工程扩展性需要同时满足。
- 如何控制幻觉：证据链接、结构化中间产物和风险审查共同约束。
- 下一步：接入更稳定的搜索 API、持久化运行历史并增加跨行业 benchmark。
